# Advanced Evolution Roadmap: Salesforce Automation Framework

This document consolidates advanced architecture ideas and high-impact engineering initiatives for the Playwright + TypeScript + Salesforce API ecosystem. The ultimate goal of these initiatives is to transform raw telemetry and execution data into business intelligence, test stability, and near-zero failure triage times.

---

## 1. Static Observability Dashboard (HTML/React SPA)
### 🎯 The Objective
Transform the raw `execution-flow.json` files generated by the semantic telemetry architecture into a rich, interactive, and highly visual report tailored for managers, developers, and QA engineers alike.

### 🛠️ Technical Approach
* **Custom Playwright Reporter:** Implement a custom class adhering to Playwright's `Reporter` interface. Within the `onEnd()` lifecycle hook, the reporter will scan the test artifact directories (`testInfo.outputDir`), gather all decentralized semantic trace logs (`FLOW`, `ACTION`, `DATA`, `SUCCESS`, `ERROR`), and unify them.
* **Static Compilation:** Develop a Node.js utility script to parse these unified JSON structures and inject the data into a responsive HTML/Tailwind CSS template, or compile a self-contained Single Page Application (SPA) using Vite/React.
* **Key Features:**
  * An interactive visual timeline correlating high-level business flows (e.g., "Creating Lead via API") directly with granular UI interactions.
  * Robust filtering capabilities by log severity and semantic telemetry categories.
  * Seamlessly embedded contextual attachments (such as screenshots and Playwright trace zip links) directly within the error step timeline.

### 💼 Business Value (Management Perspective)
Provides instant, clear visibility into the health of the test suite and release status, completely eliminating the need for non-technical stakeholders to analyze raw CI pipeline logs inside GitHub Actions.

## 2. AI-Powered Root Cause Analysis (RCA) Engine
### 🎯 The Objective
Automate the triage of test failures within the Continuous Integration (CI) pipeline by leveraging Large Language Models (LLMs) to interpret cleaned-up technical error telemetry and post actionable debugging diagnostics directly onto Pull Requests.

### 🛠️ Technical Approach
* **Conditional CI Integration:** Configure a post-execution step within the GitHub Actions workflow designed to run exclusively under the `if: failure()` condition.
* **AI-Readable Telemetry Consumption:** The script will ingest the pre-processed, structured failure context file (e.g., `error-context.md` or the localized output from the `ErrorInterpreter`).
* **Prompt Engineering & Orchestration:** Transmit the sanitized error context—stripped of distracting ANSI escape codes and redundant system noise—to an LLM API (such as GPT-4o or Gemini Pro). Instruct the model to analyze the semantic timeline leading up to the failure and categorize the root cause into:
  1. *Application Defect* (genuine unexpected UI/API behavior).
  2. *Infrastructure Instability* (network timeouts or Salesforce Sandbox latency).
  3. *Flaky Test / Automation Regression* (stale selectors or race conditions).
* **Automated PR Feedback:** Utilize the GitHub API to dynamically publish a clear, natural-language diagnostic comment on the failing branch's Pull Request.

### 💼 Business Value (Dev & QA Perspective)
Drastically minimizes the manual overhead and time spent debugging environmental noise or parsing multi-line stack traces, enabling developers to immediately address real application defects.

## 3. API-Driven Self-Healing Mechanism
### 🎯 O Objetivo
Mitigate the inherent flakiness associated with front-end automation in Salesforce Lightning components by leveraging the framework's existing robust API layer as an oracle validation checkpoint during UI execution.

### 🛠️ Technical Approach
* **Fallback Interceptor in `UiActionsComponent`:** Enhance the core interaction methods (clicks, filling forms). If a `TimeoutError` occurs during a critical UI action—such as clicking the "Save" button on an Opportunity modal—the exception block will catch it resiliently.
* **API Oracle Validation:** Before propagating a hard test failure, the framework will temporarily intercept the error and trigger an asynchronous call via the corresponding API client layer (e.g., `OpportunitiesClient.getOpportunityById()` or an optimized SOQL query). It verifies whether the record was actually successfully persisted on the Salesforce backend despite the front-end browser lag or visual rendering freezing.
* **Self-Healing Recovery Logic:**
  * *If the data is verified via API:* The test context dynamically self-heals, logs a warning regarding UI latency, updates its operational state, bypasses the surface-level UI error, and seamlessly continues execution.
  * *If the data does not exist:* The verified technical failure is safely thrown and passed on to the `ErrorInterpreter`.

### 💼 Business Value (Engineering Perspective)
Significantly reduces false-positives triggered by client-side rendering bottlenecks in Salesforce Lightning, yielding a highly reliable and trustworthy automation pipeline.

---

## 4. State-Driven Hybrid Automation (Test Isolation)
### 🎯 The Objective
Optimize overall execution velocity and stabilize complex interface validation by substituting lengthy, brittle UI click-through paths with instantaneous backend state injection via REST APIs.

### 🛠️ Technical Approach
* **Scope Isolation:** Instead of scripting a browser to manually step through 5 sequential pages (Create Account -> Create Contact -> Associate Opportunity -> Line Items) just to test a validation rule on the final closure screen, delegate the preliminary setup entirely to the API layer.
* **State Injection via API Clients:** The test script utilizes existing fluent data builders and clients (`AccountBuilder`, `ContactBuilder`, `OpportunityBuilder`) to programmatically assemble and instantly instantiate the entire prerequisite record structure in the Salesforce cloud in milliseconds.
* **Deep Linking & Session Injection:** The active Playwright browser context captures the unique IDs generated by the API setup, programmatically structures the direct record view URL (e.g., `/lightning/r/Opportunity/RECORD_ID/view`), and forces immediate deep-link navigation directly to the target UI validation page.
* **Automated Data Cleanup:** Leverage the structural `afterEach` test hooks to sequentially invoke corresponding API deletion teardowns (`deleteAccount`, `deleteOpportunity`), maintaining environment sanitization and avoiding storage clutter within the Sandbox.

### 💼 Business Value (Architecture Perspective)
Compresses end-to-end execution timelines from minutes down to seconds, strategically isolating front-end browser testing strictly to the unique visual or behavioral changes under test.

### 1. 📈 Allure Report — Visual Test Dashboard

**Impact:** Managers, PMs, Senior QAs  
**Estimated effort:** 1 day  
**Why it impresses:** Professional HTML report with execution history, embedded screenshots, and failure trends — exactly what enterprise companies expect.

#### How to implement

```bash
npm install --save-dev allure-playwright allure-commandline
```

`playwright.config.ts`:
```typescript
reporter: [
  ['list'],
  ['allure-playwright', {
    detail: true,
    outputFolder: 'allure-results',
    suiteTitle: true,
  }]
],
```

Scripts in `package.json`:
```json
"test:report": "playwright test && allure generate allure-results --clean -o allure-report && allure open allure-report"
```

`.github/workflows/playwright.yml`:
```yaml
- name: Generate Allure Report
  if: always()
  run: npm run allure:generate

- name: Upload Allure Report
  uses: actions/upload-artifact@v3
  if: always()
  with:
    name: allure-report
    path: allure-report/
```

### 2. 🗂️ Tag Strategy + Separate Pipeline per Suite

**Impact:** Tech Lead, Manager, DevOps  
**Estimated effort:** 1 day  
**Why it impresses:** Demonstrates a real quality pipeline mindset, not just isolated tests.

#### How to implement

Add tags to specs:
```typescript
test('@smoke @critical create lead with valid data', async () => { ... });
test('@regression create lead with optional fields', async () => { ... });
test('@api @smoke validate creation via API', async () => { ... });
```

`playwright.config.ts`:
```typescript
// Smoke suite
grep: /@smoke/,
```

`package.json`:
```json
"test:smoke":      "playwright test --grep @smoke",
"test:regression": "playwright test --grep @regression",
"test:api":        "playwright test --grep @api",
"test:critical":   "playwright test --grep @critical"
```

GitHub Actions with separate pipelines:
```yaml
# On PRs: smoke only
- name: Smoke Tests
  if: github.event_name == 'pull_request'
  run: npm run test:smoke

# On merge to main: full regression
- name: Full Regression
  if: github.ref == 'refs/heads/main'
  run: npm run test:regression
```

### 3. 📦 Data-Driven Testing with Dynamic Fixtures

**Impact:** Senior Dev, Senior QA  
**Estimated effort:** 1–2 days  
**Why it impresses:** Shows real coverage with multiple scenarios — happy path, edge cases, invalid fields — without duplicating code.

#### How to implement

`test-data/leads.data.json`:

```json
[
  { "scenario": "complete valid data", "firstName": "John", "lastName": "Smith", "company": "Acme", "expectedStatus": 201 },
  { "scenario": "missing company (required field)", "firstName": "Anna", "lastName": "Costa", "company": "", "expectedStatus": 400 },
  { "scenario": "invalid email", "firstName": "Peter", "lastName": "Lima", "email": "not-an-email", "expectedStatus": 400 }
]
```

`tests/api/leads/leads.create.data-driven.spec.ts`:
```typescript
import leadsData from '../../../test-data/leads.data.json';

for (const { scenario, expectedStatus, ...leadFields } of leadsData) {
  test(`create lead: ${scenario}`, async ({ leadsClient }) => {
    const lead = new LeadBuilder().withFields(leadFields).build();
    const response = await leadsClient.create(lead);
    expect(response.status).toBe(expectedStatus);
  });
}
```

### 5. 🔁 Smart Retry + Flaky Test Tagging

**Impact:** Senior QA, Tech Lead  
**Estimated effort:** Half a day  
**Why it impresses:** Mature enterprise frameworks have this. Shows you understand the real day-to-day of a QA team.

#### How to implement

`playwright.config.ts`:
```typescript
retries: process.env.CI ? 2 : 0,

// Separate projects by stability
projects: [
  {
    name: 'stable',
    grep: /^((?!@flaky).)*$/,  // all except @flaky
    retries: 1,
  },
  {
    name: 'flaky',
    grep: /@flaky/,
    retries: 3,  // more retries for unstable tests
  }
]
```

Tag unstable tests:
```typescript
test('@flaky @smoke validate login with MFA', async () => { ... });
```

### 6. 🌐 Environment Matrix in CI (sandbox vs staging)

**Impact:** DevOps, SRE, Architect  
**Estimated effort:** 1 day  
**Why it impresses:** Very few personal frameworks reach this level. Shows QA infrastructure maturity.

#### How to implement

`utils/env.ts` (expanded):
```typescript
export const ENV = {
  baseURL: process.env.SF_BASE_URL!,
  username: process.env.SF_USERNAME!,
  password: process.env.SF_PASSWORD!,
  environment: process.env.SF_ENV || 'sandbox',
};
```

`.github/workflows/playwright.yml`:
```yaml
strategy:
  matrix:
    environment: [sandbox, staging]
  fail-fast: false  # continues even if one environment fails

steps:
  - name: Run Tests on ${{ matrix.environment }}
    run: npm run test:regression
    env:
      SF_BASE_URL: ${{ secrets[format('SF_BASE_URL_{0}', matrix.environment)] }}
      SF_USERNAME: ${{ secrets[format('SF_USERNAME_{0}', matrix.environment)] }}
      SF_PASSWORD: ${{ secrets[format('SF_PASSWORD_{0}', matrix.environment)] }}
      SF_ENV: ${{ matrix.environment }}
```
